# Day 3 — Reconnaissance Detection Investigation

## Objective

Observe Windows-generated telemetry and analyze network traffic between Kali Linux and the Windows 11 VM.

## Environment

- **SOC-ATTACKER-01** — Kali Linux
- **SOC-WINDOWS-01 (Non-Prod)** — Windows 11

## Tools Used

- Nmap
- Wireshark
- Event Viewer

## Activities Performed

- Verified communication between both virtual machines
- Opened multiple applications within the Windows 11 VM to observe generated telemetry in Event Viewer
- Performed a failed login attempt to analyze corresponding Windows log activity
- Used Wireshark to capture and observe network traffic while browsing websites from the Windows VM
- Executed Nmap scans against the Windows VM to observe packet behavior and scan traffic

## Findings

- Observed different Windows events generated by application activity and failed authentication attempts
- Identified multiple protocols and observed portions of the TCP three-way handshake process in Wireshark
- Learned how retransmissions appear in packet captures and how firewall filtering impacts scan behavior
- Determined that ICMP/ping requests failed, but the Windows host remained reachable using Nmap with the `-Pn` option
- Observed that many ports appeared filtered during scanning
- Identified **TCP port 5357 as open** on the Windows host

## Lessons Learned

- Hosts may block ICMP traffic while still remaining online and reachable
- Wireshark can be used to visualize reconnaissance activity and packet flow
- Firewall configurations significantly affect host discovery and scan results
- Packet captures provide visibility into TCP behavior, retransmissions, and connection attempts
- Event Viewer generates large amounts of telemetry that require correlation and filtering

## Environment Notes

Windows 11 on UTM runs on Apple Silicon (ARM architecture).
- `Sysmon64.exe` is x64 — its kernel driver can't fully load in ARM emulation
- `Sysmon64a.exe` is the native ARM64 binary — driver loads and starts cleanly
